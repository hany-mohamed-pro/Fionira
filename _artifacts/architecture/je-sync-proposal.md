# اقتراح معماري — مزامنة القيود مع السجلات (Sync عند الرفع)

**الحالة:** معتمد ✓ (المستخدم — 2026-07-01) · **البند:** Phase B — (أ) · **التنفيذ:** لم يبدأ بعد
**النطاق:** توثيق معماري فقط. لا يُعدَّل أي كود بموجب هذه الوثيقة.

---

## 0) القرار المعماري المعتمد
> Sync عند الرفع — القيود تُولَّد وتُخزَّن في `erp_registry.json` كلما رُفعت بيانات
> جديدة، بحيث تبقى القيود المخزونة دائماً محدّثة ومطابقة للسجلات الحالية. هذا
> يحافظ على سجل المراجعة التاريخي ويُمكّن السؤال «ما الرصيد في تاريخ X؟».

## 1) اكتشاف أساسي غيّر التشخيص
مسار التفعيل الحالي (`POST /api/erp/files/lifecycle/:stagedId/activate` — `server.ts:1991`)
**لا يولّد قيوداً إطلاقاً** — يضيف السجلات فقط (`devMemoryDb.records.push(...)` عند
`server.ts:2139`) ثم `persistRegistry()`. القيود الـ1,179 المخزّنة حالياً وصلت عبر مسار
قديم `POST /api/erp/dev/sync` (`server.ts:566`، dev-only، append عند `server.ts:604`)
**لا يستدعيه العميل الحالي** (البحث في `src/` لم يُظهر أي مستدعٍ حي غير مسار
`rapid-s1/reset`).

**النتيجة:** السجلات تتحدّث مع كل تفعيل، بينما القيود مجمّدة عند لقطة قديمة → هذا هو
مصدر الـstale بالضبط (القيود المخزّنة NI ≈ 114,146 مقابل السجلات الحالية NI ≈ −116,891).

## 2) السؤال 1 — أين بالضبط يُضاف الاستدعاء؟
نقاط إضافة السجلات في `server.ts` (كل موضع `devMemoryDb.records.push`):

| السطر | المعالج | الحدث |
|---|---|---|
| **2139** | `POST /api/erp/files/lifecycle/:stagedId/activate` (1991) | **رفع ملف جديد** ← الهوك الأساسي |
| **2353** | `POST /api/erp/files/lifecycle/:stagedId/replace/:targetId` (2167) | **استبدال/تعديل ملف سابق** |
| 669 | مسار الإدخال المجمّع (`dedupedRecords`) | مسار مزامنة إدخال |
| 3259 | استعادة ملف | حالة أقل شيوعاً |
| 3584 | تصحيح سجل مفرد (`fixedRecord`) | حالة أقل شيوعاً |

**التصميم المقترح — دالة واحدة مشتركة** تُضاف في `server.ts` (لا تلمس المحرّكات):

```ts
// server.ts — helper جديد
async function syncJournalEntriesForFile(records, moduleType, fileHash, tenantId) {
  // 1) احذف قيود هذا الملف (idempotency) — نفس نمط الحذف-بالملف الموجود (server.ts:3292)
  devMemoryDb.journalEntries =
    devMemoryDb.journalEntries.filter(je => je.sourceFileId !== fileHash);
  // 2) استدعاء erp-engine (قراءة فقط — دون تعديل erp-engine.ts)
  const { generateJournalEntries } = await import('./src/backend/core/erp-engine.ts');
  const gen = generateJournalEntries(records, moduleType);
  // 3) وسم كل قيد
  gen.forEach(je => { je.tenantId = tenantId; je.sourceFileId = fileHash; je.isActive = true; });
  // 4) append
  devMemoryDb.journalEntries.push(...gen);
  // 5) persist
  persistRegistry();
}
```

تُستدعى مباشرةً بعد كل `records.push(...)` في الجدول أعلاه. تُستورَد
`generateJournalEntries` ديناميكياً تماماً كما تُستورَد `createValidationSession`
حالياً داخل نفس المعالج (`server.ts:2040`) — **استدعاء لا تعديل**.

## 3) السؤال 2 — append أم regenerate؟
`generateJournalEntries` (`erp-engine.ts:40`) **دالة نقية**: سجلات → قيود، بلا تخزين.
لا تُضيف ولا تُعيد التوليد بذاتها؛ المُستدعي هو من يقرّر. التخزين الحالي (`server.ts:604`)
**append** فقط. الأنسب لكل حالة:

- **رفع ملف جديد:** ولّد قيود سجلاته، وسمها بـ`sourceFileId`، ثم **append**.
- **تعديل/استبدال ملف سابق:** **احذف قيود الملف القديم بـ`sourceFileId` ثم ولّد وأضف
  قيود البديل** — أي **إعادة توليد على مستوى الملف الواحد** لا الكل.

**ليس** regenerate-all عالمياً، لأنه (أ) يمحو أي تعديلات يدوية على القيود (زر
`isActive`، `server.ts:1138`)، (ب) يهدر معالجة ملفات لم تتغيّر. التجزئة بـ`sourceFileId`
هي الحبيبة الصحيحة، ومطابقة لآلية الحذف-بالملف الموجودة.

## 4) السؤال 3 — خطر التكرار (نفس الملف مرتين)
**نعم، خطر قائم** إن أضفنا دون تنظيف. المنع بطبقتين:

- **idempotency على مستوى الملف:** الخطوة 1 في الدالة أعلاه
  (`filter(je => je.sourceFileId !== fileHash)` قبل الإضافة) — **نفس النمط الموجود
  فعلاً** في حذف الملف (`server.ts:3292`). فتفعيل/استبدال نفس الملف يستبدل قيوده ولا
  يكرّرها (عملية idempotent).
- **dedup الموجود مسبقاً:** تصنيف الـstaged-upload بالـ`fileHash`
  (NEW_PERIOD_SOURCE / CORRECTED_VERSION / OVERLAP) يمنع أصلاً رفع الملف المطابق.

## 5) السؤال 4 — الأثر على ميزان المراجعة / دفتر الأستاذ / RealBalanceSheet
**تستفيد الثلاثة تلقائياً — صفر تعديل في أيٍّ منها.** جميعها تقرأ نفس مصدر
`devMemoryDb.journalEntries` (عبر `/api/debug/journalEntries/raw` و`/api/erp/ledger`).
بمجرّد أن يولّد التفعيل قيوداً محدّثة من السجلات الحالية، تعكس الثلاثة البيانات الحالية
فوراً. **هذه أناقة الحل:** إصلاح المصدر يُصلح TrialBalance + GeneralLedger + الميزانية
معاً — ومطابقة RealBalanceSheet مع `computePnLCore` ستتحقّق (ضمن فرق الـ54.78 المعروف،
البند 13).

## 6) السؤال 5 — تقييم المخاطر
- **يمسّ `categorization-engine.ts`؟ لا** — `generateJournalEntries` تقرأ حقل `Category`
  الموجود على السجل (المُحدَّد وقت الإدخال)، لا تُصنّف.
- **يمسّ `erp-engine.ts`؟ لا** — يُستدعى فقط (ثبت في Phase A أنه ينتج قيوداً مطابقة
  للسجلات ضمن 54.78).
- **التعقيد: أقل مما توقعنا** — دالة واحدة عند نقاط الإضافة الموجودة؛ TB/GL/BS تُصلَح
  تلقائياً؛ نمط الحذف-بالملف جاهز لإعادة الاستخدام.
- **⚠️ الخطر الحقيقي — القيود الـ1,179 القديمة:** عند أول تفعيل-مع-توليد، ستُضاف قيود
  جديدة للملفات الجديدة بينما تبقى القديمة → **ازدواج حسابي للملفات القديمة**. لذا يلزم
  **backfill لمرة واحدة: إعادة توليد كل القيود من السجلات الحالية دفعةً واحدة** (تستبدل
  الـ1,179)، ثم التوليد التزايدي لكل ملف لاحقاً. هذه **هجرة بيانات مقصودة** (تكتب
  `erp_registry.json`) ببوابة قبول = مطابقة الهللة (الأصول = الالتزامات + حقوق الملكية،
  والربح يطابق `computePnLCore` ضمن 54.78).
- **تعديلات القيود اليدوية** (`isActive`، `server.ts:1100-1141`): إعادة توليد ملفٍ ما
  تلغي تعديلاته اليدوية لذلك الملف. تخفيف: التوليد فقط عند activate/replace لذلك الملف
  (لا عند عمليات أخرى)، أو الحفاظ على أعلام `isActive` عند التوليد. مُنبَّه إليه.
- **مسار Postgres** (`isConnected()`): لا يوجد جدول قيود بعد (القيود dev فقط) —
  يُعالَج منفصلاً لاحقاً.

## 7) التوصية — تقسيم Phase B-أ إلى خطوتين مؤكَّدتين
- **B-أ-1 (backfill لمرة واحدة):** سكربت/endpoint يعيد توليد كل القيود من السجلات
  الحالية مرة واحدة، ببوابة مطابقة الهللة، يستبدل الـ1,179 القديمة.
- **B-أ-2 (المزامنة التزايدية):** الدالة المشتركة `syncJournalEntriesForFile` عند
  activate/replace (وبقية نقاط الإضافة)، idempotent بـ`sourceFileId`.

كل خطوة بموافقة صريحة منفصلة، مع إثبات حي حقيقي قبل أي commit نهائي.
